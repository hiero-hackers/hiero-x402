## The flow in one paragraph

A client requests a resource. The server answers **HTTP 402** with a JSON body
listing acceptable payments (`accepts[]`). The client constructs and signs a
payment authorization, base64-encodes it, and retries the request with the
payment in a header. The server hands payload + requirements to a
**facilitator** (`POST /verify`, then `POST /settle`); the facilitator checks
the authorization, submits it on-chain, and the server returns **200** with the
resource plus a base64 settlement receipt in a response header.

Three roles: **resource server** (sells the API), **client** (pays — human or
agent), **facilitator** (verifies + settles on-chain so the resource server
never touches keys).

## Core types (transport- and chain-independent)

### 402 challenge body — `PaymentRequirementsResponse`

```json
{
  "x402Version": 1,
  "error": "X-PAYMENT header is required",
  "accepts": [
    {
      "scheme": "exact",
      "network": "base-sepolia",
      "maxAmountRequired": "10000",
      "asset": "0x036C…",
      "payTo": "0x2096…",
      "resource": "https://api.example.com/premium-data",
      "description": "Access to premium market data",
      "mimeType": "application/json",
      "maxTimeoutSeconds": 60,
      "extra": {}
    }
  ]
}
```

- Amounts are **strings in atomic units** (our house rule already).
- `extra` is the scheme-specific escape hatch — on Hedera it carries the
  fee payer (see [[02-hedera-mapping]]).
- **v1 vs v2 naming**: the v1 spec says `maxAmountRequired`; the current
  packages (`@x402/core` 2.x) and the official Hedera scheme spec use v2 with
  plain `amount`. Build against v2, accept both when parsing.

### `PaymentPayload` (the retried request's payment)

v1 (EVM example): `{ x402Version, scheme, network, payload: { signature, authorization } }`
where `authorization` is EIP-3009 fields (`from/to/value/validAfter/validBefore/nonce`).

v2 (what the Hedera scheme uses) wraps more context:

```json
{
  "x402Version": 2,
  "resource": { "url": "…", "description": "…", "mimeType": "…" },
  "accepted": { …the chosen PaymentRequirements… },
  "payload": { …scheme-specific… }
}
```

The `payload` member is the only chain-specific part — per-scheme specs define
it (Hedera: a single base64 field, see [[02-hedera-mapping]]).

### `SettlementResponse`

```json
{ "success": true, "transaction": "0x…", "network": "base-sepolia", "payer": "0x…" }
```

`errorReason` appears on failure (`transaction` becomes `""`). On Hedera the
field is `transactionId` in `0.0.x@seconds.nanos` form — exactly what a
mirror-node lookup (and hiero-receipts) wants.

## HTTP transport — the headers

| Purpose                   | v1 header            | v2 header (current packages)                        |
| ------------------------- | -------------------- | --------------------------------------------------- |
| Challenge (with the 402)  | body only            | `payment-required` (base64 JSON; body also allowed) |
| Payment on retry          | `X-PAYMENT`          | `payment-signature`                                 |
| Settlement receipt on 200 | `X-PAYMENT-RESPONSE` | `payment-response`                                  |

All header values are **base64-encoded JSON** of the types above. The scaffold
template's server reads `payment-signature` _or_ `x-payment` — do the same for
compatibility. Status mapping: 402 = payment required or failed, 400 =
malformed payload, 200 = paid + served; a failed settlement returns 402 with a
`success:false` receipt header.

## Facilitator API (HTTP, standardized)

- `POST /verify` — body `{ x402Version, paymentPayload, paymentRequirements }`
  → `{ isValid, payer }` or `{ isValid: false, invalidReason }`. **No chain
  write** — pure validation (+ simulation/preflight).
- `POST /settle` — same body → executes on-chain →
  `{ success, transaction|transactionId, network, payer }`.
- `GET /supported` — `{ kinds: [{ x402Version, scheme, network }] }`. On
  Hedera implementations this also advertises the **feePayer** the facilitator
  sponsors, which resource servers fold into `extra`.

Standard `invalidReason` / `errorReason` codes worth reusing verbatim:
`insufficient_funds`, `invalid_scheme`, `invalid_network`, `invalid_payload`,
`invalid_payment_requirements`, `invalid_transaction_state`,
`unexpected_verify_error`, `unexpected_settle_error`.

## Security model (spec §10)

- Replay: EVM leans on EIP-3009 nonces + validity windows. Hedera gets replay
  protection **for free** from the network — a `TransactionId` can only be
  submitted once and has a ~2-minute validity window.
- The facilitator is the trust anchor: it must prove the payment does exactly
  what the requirements say **before** sponsoring/submitting it. The
  per-scheme MUST-rules (Hedera's are strict and enumerated) are the real
  security content of the protocol.
